## What Was Fixed

The smoke run exposed a small set of real pipeline issues that are now fixed in the working tree:

1. Wildcard disambiguation in the top-level Snakefile.
   - Added explicit wildcard constraints for `experiment` and `well_id` so Snakemake does not greedily match per-well paths as experiment-level targets.

2. Segmentation contract declaration.
   - `segment_and_track_well` now declares `segmentation_tracking.csv` as an explicit output, not just a side effect.
   - This unblocks downstream snip processing and stage prediction rules that consume that file directly.

3. Stage prediction sentinel separation.
   - Per-well stage prediction jobs now emit `.stage_predictions.computed`.
   - Experiment-level validation still emits `.stage_predictions.validated`.
   - This removes ambiguity between per-well compute and merged validation rules.

4. Plate metadata schema relaxation.
   - `treatment` is no longer required in `plate_metadata`.
   - The smoke experiments available for this run do not consistently populate that field.
   - `start_age_hpf` and `temperature` remain required for stage predictions.

5. Stage prediction merge robustness.
   - `compute_stage_predictions.py` now handles `well_index` safely after joining segmentation outputs to `frame_contract.csv`.
   - This fixes the `KeyError: 'well_index'` caused by pandas suffixing overlapping column names during the merge.

## What We Learned From The Run

The first actual runtime blocker was not a code traceback. `materialize_stitched_images` was being killed with exit code `137`, which indicates the process was OOM-killed. Re-running the same YX1 materialization step in a higher-memory login session resolved that immediately.

That means the pre-segmentation path is wired correctly, but YX1 stitched-image materialization is still memory-hungry enough that it may fail in lower-memory environments.
